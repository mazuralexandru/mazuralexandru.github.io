643. Reset & Named vs. Numeric Entities

We've covered the reasons for using entities (escaping, symbols, whitespace control). Now let's compare the two syntax forms: Named vs. Numeric Character References.

Resetting the Page:

To focus on this comparison, we will remove the non-breaking space examples.

Named Character References (&name;)

  • Syntax: &entityname; (e.g., &copy;, &lt;, &euro;)
  • Pros:
    • Readability: Much easier for humans to read and understand the intended character directly from the source code.
    • Memorability: Common entity names are often easier to remember than numeric codes.
  • Cons:
    • Limited Set: Named entities only exist for a specific (though large in HTML5) subset of Unicode characters. Many less common characters or symbols do not have named entities.
    • Case-Sensitive: Names must be typed exactly correctly (&Eacute; vs &eacute;).

Numeric Character References (&#number; or &#xhex;)

  • Syntax: &#DecimalNumber; (e.g., &#169;) or &#xHexNumber; (e.g., &#xA9;)
  • Pros:
    • Universality: Can represent any character in the entire Unicode standard by using its unique code point number.
  • Cons:
    • Readability: Much harder for humans to read the source code and immediately know which character is represented without looking it up.
    • Typo-Prone: Easier to make mistakes when typing numeric codes.

Recommendation:

  1. Use UTF-8: Prioritize direct character input with UTF-8 encoding for most non-reserved characters.
  2. Escape Reserved: Use named entities (&lt;, &gt;, &amp;, &quot;, &apos;) for the essential reserved HTML characters.
  3. Common Symbols: Use named entities (&copy;, &reg;, &euro;, &mdash;, etc.) for commonly needed symbols where readability is helpful.
  4. Rare Characters: Only resort to numeric entities if a character lacks a named entity and you cannot (or choose not to) use the direct UTF-8 character in your source file for some specific reason.
