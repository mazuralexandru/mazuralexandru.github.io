691. Accessibility (A11y): Role of HTML (Semantic Foundation)

Let's explicitly connect the dots: why is semantic HTML, which we focused on in Module 10, the absolute foundation for web accessibility?

HTML Provides Structure and Meaning for AT:

Assistive Technologies (AT) do not "see" the visual layout of a page created by CSS. Instead, they primarily interact with the Document Object Model (DOM) tree that the browser builds from your HTML source code. The tags and attributes you use in your HTML are the primary source of information for AT.

  • Landmarks (<header>, <nav>, <main>, etc.): Tell AT about the major regions, enabling efficient navigation. <div> provides no such information.
  • Headings (<h1>-<h6>): Create a navigable outline and indicate content hierarchy. <p> styled to look bold/large does not.
  • Lists (<ul>, <ol>, <li>): Announce list structure and item count/position. Manual bullets/numbers in <p> tags do not.
  • Tables (<table>, <th>, <td>, <caption>, scope): Define data relationships crucial for understanding tabular information. <div>-based grids do not inherently convey these relationships.
  • Forms (<label>, <fieldset>, <legend>): Programmatically associate labels and group related controls. Essential for understanding and filling forms. Placing text next to an input without <label for="..."> provides no link for AT.
  • Images (<img> with alt): Provide text descriptions for visual content.
  • Links (<a>): Announced as interactive links; descriptive text is key. Contrast with <span onclick="..."> which isn't announced as a link by default.
  • Emphasis/Importance (<em>, <strong>): Can potentially influence screen reader intonation.
  • Language (lang): Enables correct pronunciation.

Semantic HTML = Accessibility Built-In (Often):

By using standard HTML elements according to their intended meaning, you automatically gain a significant amount of built-in accessibility. Browsers and AT are designed to understand these standard semantics.

Non-Semantic HTML = Accessibility Barriers:

Relying heavily on non-semantic elements (<div>, <span>) forces you to manually add accessibility information back in using ARIA attributes, which is:

  • More complex.
  • More error-prone.
  • Often less robust than native HTML semantics.
  • Still requires correct JavaScript implementation for keyboard interaction on custom controls.

The First Rule of ARIA Use: Use native HTML elements and attributes whenever they provide the required semantics and behavior before resorting to ARIA.

✅ Observation: No code changes. Review the semantic elements learned in previous modules. Understand that using <nav> instead of <div class="nav">, or <h1> instead of <p class="title">, directly provides essential structural and semantic information needed by assistive technologies.

✨ Key Takeaway: Correctly using semantic HTML elements is the most important first step towards building accessible websites. These elements provide the necessary structure and meaning for assistive technologies to interpret and present content effectively.
