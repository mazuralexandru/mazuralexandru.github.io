702. Accessibility: ARIA Basics - The 5 Rules of ARIA Use

Because ARIA modifies native semantics, using it incorrectly can actually harm accessibility more than help. The W3C provides essential guidelines known as the "Rules of ARIA Use". Understanding these is crucial before applying ARIA attributes.

The 5 Rules (Paraphrased):

  1. (Use Native HTML First) If you can use a native HTML element (<button>, <input type="checkbox">, <nav>, <h1>, etc.) or attribute (required, disabled, <label for="...">) that already provides the semantics and behavior you need, do that instead of using ARIA. Native HTML has built-in accessibility and keyboard support. Don't re-invent a button with <div role="button"> if <button> works. Use ARIA primarily as a fallback when native HTML is insufficient for the pattern you're building.

  2. (Don't Change Native Semantics) Do not add ARIA roles or attributes in a way that conflicts with the element's default, implicit semantics. For example, don't put role="button" on an <h1> or role="heading" on a <a>. This creates confusing and conflicting information for assistive technologies.

  3. (Ensure Keyboard Accessibility) All interactive ARIA controls (e.g., anything with role="button", role="slider", role="tab") must be navigable and operable using only the keyboard. ARIA adds the semantics, but you must implement the keyboard interaction logic (handling Enter, Space, arrow keys, etc.) using JavaScript if you're building custom controls.

  4. (Don't Hide Focusable Elements) Do not use role="presentation" (which removes semantics) or aria-hidden="true" (which hides from AT) on an element that is currently visible and focusable, or that contains focusable descendants. Users must be able to perceive and interact with elements that can receive focus.

  5. (Provide Accessible Names) All interactive elements (buttons, links, inputs, custom controls) must have an accessible name. This name can come from the element's content (e.g., text inside <button>), an associated <label>, alt text (for image inputs), or explicitly via aria-label or aria-labelledby.

Mantra: "No ARIA is better than Bad ARIA." Incorrect ARIA can create more barriers than it solves. Use it carefully, understand its purpose, and always test with assistive technologies.

✅ Observation: No code changes. Carefully review these five rules. Rule #1 (Use Native HTML First) is the most important guiding principle. Rule #3 (Keyboard Accessibility) highlights the link between ARIA semantics and required JavaScript behavior for custom controls. Rule #5 reinforces the need for labels/names.

✨ Key Takeaway: Follow the 5 Rules of ARIA Use: Prioritize native HTML, don't change native semantics, ensure keyboard accessibility for custom controls, don't hide focusable elements, and provide accessible names for all interactive elements. "No ARIA is better than Bad ARIA."
