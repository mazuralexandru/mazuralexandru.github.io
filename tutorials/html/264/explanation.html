264. Attributes: Syntax Recap

Let's solidify the syntax rules for writing attributes.

Core Syntax (name="value")

  • Placement: Attributes are always placed within the opening tag of an element, after the element name.
    • Correct: <p lang="en">
    • Incorrect: <p> lang="en"</p> or <p lang="en"></p lang="en">
  • Structure: Typically consists of the attribute name, followed by an equals sign (=), followed by the attribute value enclosed in quotes.
  • Whitespace:
    • Space(s) must separate the element name from the first attribute, and between multiple attributes.
      • Example: <img src="..." alt="..." width="...">
    • Whitespace around the equals sign (name = "value") is technically allowed but unconventional. Standard practice is name="value".
  • Value Quoting:
    • Double Quotes ("value"): This is the most common and recommended convention. It's clear and handles most values easily.
      • Example: class="error-message"
    • Single Quotes ('value'): Also valid. Useful if the attribute value itself needs to contain double quotes.
      • Example: onclick='alert("Hello!");' (Though inline JS is generally discouraged).
    • Unquoted Values (Discouraged): Technically valid only if the value contains no whitespace, ", ', =, <, >, or ` characters. This is considered bad practice as it's error-prone and less readable. Always use quotes.
      • Example (Valid but Bad): <input type=text>
      • Example (Invalid): <div class=error message> (Space requires quotes).

Boolean Attributes

  • Syntax: For attributes that represent a true/false state, you only need to include the attribute name; its presence implies true.
    • Example: <input type="checkbox" checked> (The checked attribute is present).
    • Example: <input type="checkbox"> (The checked attribute is absent, implying false).
  • Historical Note: XHTML required boolean attributes to be written as name="name" (e.g., checked="checked"). HTML5 simplifies this to just the name.

Case Sensitivity

  • Attribute Names: Are case-insensitive in HTML (e.g., CLASS works like class). Best Practice: Always use lowercase for consistency and compatibility.
  • Attribute Values: Sensitivity depends on the specific attribute. id and class values are case-sensitive. Keyword values (like _blank, en) are often case-insensitive, but following the standard casing (e.g., lowercase language codes) is recommended.

✅ Observation: No code changes. This is a review of syntax rules learned implicitly before. Ensure you understand where attributes go, the name="value" structure, and the importance of quoting values.

✨ Key Takeaway: Place attributes in the opening tag, use name="value" format, always quote values (preferably double quotes), use lowercase for attribute names, and understand boolean attribute syntax.
