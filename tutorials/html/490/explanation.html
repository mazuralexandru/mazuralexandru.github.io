490. Module 9 Recap: Forms - Collecting User Input

This module provided a comprehensive introduction to HTML forms, the essential mechanism for collecting user data on the web.

Key Concepts Reviewed:

  • Purpose: Forms collect user input for server-side processing (logins, searches, contacts, uploads, etc.).
  • <form> Element:
    • The container for form controls.
    • action: URL where data is sent (server-side endpoint).
    • method: HTTP method (GET or POST).
      • GET (Default): Data in URL (query string), visible, length limits, idempotent (searches). Never for sensitive data.
      • POST: Data in request body, hidden from URL, no length limit, not idempotent (logins, updates, uploads). Required for sensitive data/files.
    • enctype (for POST): Data encoding type.
      • application/x-www-form-urlencoded (Default): Encodes spaces/special chars. For text data.
      • multipart/form-data: Required for <input type="file">. No value encoding.
      • text/plain: Debugging only.
    • name: Form identifier (legacy JS).
    • target: Where response loads (_self, _blank, etc.).
    • novalidate: Disables native browser validation.
    • autocomplete (on form): Sets default autofill (on/off).
    • No Nesting: Forms cannot be nested.
  • <input> Element (Void): The workhorse for various controls.
    • type [Required]: Defines control type (text, password, email, number, url, tel, date, time, datetime-local, month, week, search, checkbox, radio, file, hidden, submit, reset, button, image, color, range).
    • name [Required for Submission]: Key for submitted name=value pair.
    • value: Meaning depends on type: initial value (text types), submitted value (checkbox/radio), button label (input buttons).
    • id [Required for <label>]: Unique identifier for label association.
  • Selection Inputs:
    • type="checkbox": Zero or more selections. Needs name and value. checked for default.
    • type="radio": Exactly one selection from a group. Requires shared name and unique value. checked for default (on one).
  • Labels (<label>):
    • Essential for accessibility and usability. Provides description and increases click target.
    • Explicit Association (for/id) [Recommended]: <label for="ctrlId"> matches <input id="ctrlId">. Flexible, robust.
    • Implicit Association (Wrapping): <label>Text <input></label>. Less flexible/robust.
  • Other Controls:
    • <textarea>: Multi-line text input. Content between tags is initial value. Use CSS for sizing.
    • <select>: Dropdown list container.
    • <option>: Defines choices within <select>. value attribute is submitted value; text content is displayed label. selected for default.
    • <optgroup>: Groups <option>s under a label within <select>.
  • Buttons:
    • <input type="submit"> / <button type="submit">: Trigger form validation & submission. <button> preferred (allows HTML content).
    • <input type="reset"> / <button type="reset">: Reset form to initial values. Discouraged due to usability risks.
    • <input type="button"> / <button type="button">: Generic button, no default action. Requires JavaScript. Specify type="button" on <button> to prevent default submit behavior.
  • File Inputs (<input type="file">): Requires method="POST" and enctype="multipart/form-data" on <form>. accept hints file types, multiple allows multi-select. Server validation crucial.
  • Hidden Inputs (<input type="hidden">): Submits non-visible name=value pairs (e.g., IDs, tokens). Not secure for sensitive secrets.
  • Structure (<fieldset>, <legend>): Group related controls semantically. <legend> provides accessible group label (must be first child of <fieldset>). Essential for radio/checkbox groups.
  • Validation Attributes:
    • required: Mandatory input.
    • disabled: Unusable, not submitted, not validated.
    • readonly: Not editable, IS submitted, IS validated.
    • placeholder: Hint only, disappears, not a label substitute.
    • pattern: Regex validation for text types (use title for hint).
    • min/max/step: Range/interval validation for numeric/date types.
    • minlength/maxlength: Character length validation for text/textarea.
    • autocomplete (on controls): Fine-grained autofill hints (tokens like given-name, email, new-password, off).
  • Native Validation: Browser checks constraints on submit (unless novalidate). Blocks submission on failure, shows default UI. Supplement/override with CSS/JS for better UX, but always validate server-side.
