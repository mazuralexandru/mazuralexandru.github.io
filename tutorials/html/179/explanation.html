179. Security Best Practice: rel="noopener noreferrer"

We've seen rel="noopener" (prevents window.opener access) and rel="noreferrer" (prevents sending the Referer header and implies noopener). What's the best practice when using target="_blank"?

Combine Both: rel="noopener noreferrer"

The widely recommended best practice is to include both values together in the rel attribute, separated by a space:

  • Syntax: rel="noopener noreferrer"

Why Combine Them?

  1. Clarity: Explicitly states both intents – preventing opener access (noopener) AND omitting the referrer (noreferrer).
  2. Maximum Compatibility: While noreferrer implies noopener in modern browsers, older browsers might not support this implication fully. Including both ensures the noopener behavior works even on slightly older browsers that might still support noopener but not the implication from noreferrer.
  3. Security & Privacy: Provides the strongest combination of security (mitigating tabnabbing) and privacy (hiding the referring URL).

Unless you have a specific reason to send the Referer header for external links opening in a new tab (which is rare), using rel="noopener noreferrer" is the safest default.

Code Change:

We have updated the rel attribute on our link in index.html to include both noopener and noreferrer.

<p>
  Example Link (Best Practice):
  <a href="https://developer.mozilla.org/" target="_blank" rel="noopener noreferrer"> <!-- Combined noopener and noreferrer -->
    MDN Home (New Tab, Safest)
  </a>
</p>

✅ Observation:

  1. Note the combined rel="noopener noreferrer" attribute in the <a> tag.
  2. Clicking the link still opens MDN in a new tab. The behavior is visually unchanged, but this combination provides the strongest security and privacy posture.

✨ Key Takeaway: Best Practice: When using target="_blank", always add rel="noopener noreferrer" to the <a> tag for maximum security, privacy, and cross-browser compatibility.
