693. Accessibility (A11y): Content Element Semantics Review

Let's review how other semantic content elements contribute to accessibility.

1. Lists (<ul>, <ol>, <li>, <dl>, <dt>, <dd>) Recap:

  • Accessibility Impact:
    • Screen readers announce the start and end of lists, the list type (list, ordered list, description list), and often the number of items.
    • Users can navigate item by item.
    • Ordered lists announce the item number (e.g., "1. First step").
    • Description lists clearly associate terms (<dt>) with their descriptions (<dd>).
  • Failure Example: Creating lists using <p> tags with manual bullets (*) or numbers (1.) provides no structural information; screen readers just see paragraphs.

2. Tables (<table>, <caption>, <th>, <td>, scope) Recap:

  • Accessibility Impact:
    • <caption> provides the table's accessible name.
    • <th> identifies header cells.
    • scope="col" / scope="row" programmatically links header cells to data cells. Screen readers use this to announce the relevant headers when reading a data cell, providing context.
  • Failure Example: Using <div>s and CSS to create a visual grid provides no semantic relationships for screen readers. Using <table> for layout forces a nonsensical reading order. Forgetting scope makes data cell context ambiguous.

3. Links (<a>) Recap:

  • Accessibility Impact:
    • Announced as "Link".
    • Descriptive link text is crucial (Module 4) because users often navigate by listing links out of context. "Click here" is unhelpful.
  • Failure Example: Using generic link text. Using non-link elements (<span> with onclick) styled as links - these aren't announced as links and aren't keyboard focusable/activatable by default.

4. Emphasis/Importance (<em>, <strong>) Recap:

  • Accessibility Impact: May influence screen reader voice inflection (variable support). Clearly conveys semantic meaning regardless of visual styling.
  • Failure Example: Using <i>/<b> or CSS font-style/font-weight when semantic emphasis/importance is intended – the meaning is lost for AT.

5. Other Elements (<abbr>, <time>, <address>, etc.) Recap:

  • Provide specific semantic meaning that can be interpreted by AT (e.g., <abbr title="..."> for expansion, <time datetime="..."> for unambiguous date/time).

✅ Observation: No code changes. Review the semantic elements for lists, tables, links, and inline text. Reinforce how using the correct tag provides built-in accessibility information that non-semantic alternatives lack.

✨ Key Takeaway: Properly using semantic elements for lists, tables, links, emphasis, and other specific content types provides essential context and structure for assistive technologies, making the content more understandable and navigable.
