345. Common <link> Types: Linking Favicons (rel="icon") - Behavior & Interactions

How do browsers find and use the favicons you declare?

Browser Discovery Process:

  1. HTML <head> Scan: The browser first parses the <head> section looking specifically for <link> tags with rel="icon" (and related values like apple-touch-icon, shortcut icon, mask-icon).
  2. Selection Logic: If multiple icons are declared, the browser uses the type, sizes, and media attributes (and sometimes internal heuristics) to select the most appropriate icon for the specific context where it needs to be displayed (e.g., a 32x32 PNG for a high-resolution tab, a 16x16 version for bookmarks, the apple-touch-icon for iOS home screens). SVG icons (sizes="any") might be preferred if available due to scalability.
  3. Root Fallback (/favicon.ico): If the browser finds no suitable <link rel="icon"> tags in the <head>, it will often make a separate request to the website's root directory to see if a file named favicon.ico exists there. If found, it will use that as a fallback. Relying solely on this fallback is not recommended; always declare icons explicitly.

Display Locations:

Where the chosen favicon appears depends on the browser and context:

  • Browser Tabs
  • Address Bar (less common now)
  • Bookmarks / Favorites Menu & Bar
  • Browser History
  • Desktop Shortcuts (sometimes)
  • Mobile Home Screen Icons (for apple-touch-icon or via Web App Manifest)
  • Search Engine Results Pages (SERPs) - Google sometimes shows favicons next to results.
  • Social Media Link Previews (sometimes)

Caching:

  • Favicons are often cached aggressively by browsers because they are assumed not to change frequently.
  • If you update your favicon file(s), users might not see the change immediately. Techniques to force updates include:
    • Clearing the browser cache (user action).
    • Renaming the icon file and updating the href in the <link> tag.
    • Adding a version query string to the href (e.g., href="/favicon.ico?v=2"), though cache-busting effectiveness varies.

✅ Observation: No code changes. Understand the process: browsers prioritize icons declared via <link> tags in the <head>, using attributes like sizes and type to choose the best fit, and may fall back to checking /favicon.ico. Recognize that favicons appear in many places beyond just the tab and are often heavily cached.

✨ Key Takeaway: Explicitly declare favicons using <link> tags in the <head> for reliability. Browsers select the best match based on context and attributes, falling back to /favicon.ico if needed. Be aware of caching when updating icons.
