317. display Value: inline

Files Modified: index.html, style.css

Elements with display: inline; behave very differently from block elements.

  • Behavior:
    1. Flows with Text: Inline elements sit within lines of text or alongside other inline elements. They do not force line breaks before or after themselves.
    2. Width/Height Ignored: Explicit width and height properties are ignored. The element's size is determined by its content.
    3. Limited Vertical Spacing: Vertical padding, border, and margin are applied visually but generally do not affect the height of the line box or push surrounding lines away vertically (they might overlap). Horizontal padding, border, and margin are respected and affect horizontal spacing.
  • Default Inline Elements: <span>, <a>, <em>, <strong>, <i>, <b>, <img> (replaced inline), <input> (replaced inline), <br>, <code>, <mark>, etc.

✅ Observe index.html:

  • We've added a paragraph containing several <span> elements, which are inline by default.

✅ Switch to style.css:

  • We've removed the previous block styles.
  • We've added styles for the .inline-span class. Note that we try to set width, height, and vertical margin/padding.
/* Display Inline Example */
.inline-span {
  background-color: rgba(255, 192, 203, 0.4); /* Faint pink */
  border: 1px solid pink;
  padding: 10px; /* Vertical padding might overlap */
  margin: 10px; /* Vertical margin ignored for line height */
  width: 200px; /* Ignored */
  height: 50px; /* Ignored */
}

✨ Observe the Preview:

  • The spans appear inline within the paragraph text, flowing one after another on the same line until wrapping naturally.
  • The pink background and border are applied.
  • The horizontal padding and margin (10px each side) create space between the spans horizontally.
  • The vertical padding (10px top/bottom) is visible on the background/border but likely overlaps with the lines above/below it without increasing the line height.
  • The vertical margin (10px top/bottom) has no effect on the spacing between lines.
  • The explicit width and height are completely ignored; the spans size themselves based on their text content ("Span 1", "Span 2", etc.).
