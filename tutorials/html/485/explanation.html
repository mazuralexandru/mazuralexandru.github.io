485. Forms: Validation & Assistance - autocomplete (on Controls)

We saw the autocomplete attribute on the <form> element sets a default (on or off). However, applying it to individual form controls provides much more granular control and better hints for browser autofill features.

autocomplete on Controls:

  • Purpose: Allows you to specify what kind of data the input field expects, enabling browsers and password managers to autofill the field more accurately and securely. It can also be used to override the form-level setting for specific fields.
  • Applies To: <input> (most types), <select>, <textarea>.
  • Syntax: autocomplete="value"

Key Values:

  1. on / off: Explicitly enable or disable autofill for this specific field, overriding the <form> default if present. Use off sparingly (e.g., for one-time codes, CAPTCHA inputs, or critical security fields where autofill might pose a risk).

  2. Autofill Detail Tokens (Highly Recommended): These provide specific hints about the expected data type. Using these tokens is generally better than just using on, as it helps password managers distinguish fields correctly (e.g., differentiating a username from an email, or a current password from a new one) and improves the accuracy of autofill suggestions.

    • Common Examples (Many more exist - see HTML Standard):
      • name, honorific-prefix, given-name, family-name
      • email
      • username
      • new-password (Hint for password managers during sign-up/change)
      • current-password (Hint for password managers during login)
      • street-address, address-line1, postal-code, country
      • tel
      • bday, bday-day, bday-month, bday-year
      • cc-name, cc-number, cc-exp, cc-csc (For payment forms - use with care and always over HTTPS)

Code Change:

We've added inputs demonstrating specific autocomplete tokens (given-name, email) and explicitly turning it off for a sensitive field, overriding the form-level default (which we've set back to on for contrast, though on is the implicit default anyway).

<!-- index.html snippet -->
<form action="/submit-data" method="POST" autocomplete="on"> <!-- Form level 'on' -->
  <!-- Removed length examples -->
  <p>
    <label for="fname">First Name:</label>
    <input type="text" id="fname" name="fname" autocomplete="given-name"> <!-- Specific token -->
  </p>
   <p>
    <label for="email">Email:</label>
    <input type="email" id="email" name="email" autocomplete="email"> <!-- Specific token -->
  </p>
   <p>
    <label for="cc_num">Credit Card:</label>
    <!-- Turning off for sensitive field -->
    <input type="text" id="cc_num" name="cc_num" autocomplete="off"> <!-- Explicitly off -->
  </p>
  <p><button type="submit">Submit</button></p>
</form>

✅ Observation:

  1. Examine the autocomplete attributes on the <form> and the individual <input> tags.
  2. In the preview, focus the "First Name" or "Email" fields. Your browser (if it has saved data) should offer relevant autofill suggestions based on the specific tokens (given-name, email).
  3. Focus the "Credit Card" field. The browser should ideally not offer autofill suggestions because autocomplete is explicitly set to off for this specific field, overriding the form's on.

✨ Key Takeaway: Use the autocomplete attribute on individual form controls with specific detail tokens (like given-name, email, new-password, off) to provide fine-grained hints for browser autofill, improving accuracy and security over just using the form-level on/off. Use off sparingly.
