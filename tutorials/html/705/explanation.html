705. Module 15 Recap: HTML Accessibility Fundamentals

This module introduced the core concepts of Web Accessibility (A11y) and highlighted how fundamental HTML practices contribute to creating inclusive web experiences.

Key Concepts Reviewed:

  • What is Accessibility?: Designing/developing so people with disabilities (visual, auditory, motor, cognitive) can perceive, operate, understand, and interact with the web effectively.
  • Why It Matters: Ethical imperative (inclusion), legal requirements (ADA, EAA, WCAG), business benefits (wider audience, SEO, brand image), technical quality.
  • Assistive Technologies (AT): Screen readers, magnifiers, keyboard-only, voice control, switch devices. AT relies on well-structured HTML.
  • Semantic HTML is Foundation: Using elements for their meaning (<nav>, <h1>, <ul>, <table>, <label>, etc.) provides inherent structure and context crucial for AT. Prefer native elements over non-semantic <div>/<span> + ARIA.
  • Content Accessibility Reviews:
    • Headings (<h1>-<h6>): Provide navigable outline; use hierarchically, don't skip levels.
    • Landmarks (<header>, etc.): Enable quick navigation between page regions.
    • Lists (<ul>, <ol>, etc.): Announce structure and allow item navigation.
    • Images (alt): Critical for non-text content. Describe info/function or use alt="" for decorative.
    • Forms (<label>, <fieldset>, <legend>): Essential for understanding and operating controls. Use for/id. Group related items.
    • Tables (<caption>, <th>, scope): Essential for understanding data relationships. Use for data only.
  • Language (lang): Ensures correct pronunciation by screen readers. Declare on <html> and specific elements for language changes.
  • Keyboard Navigation:
    • Requirement: All functionality must be keyboard operable.
    • Focus: Must be visible (:focus-visible, avoid outline: none without replacement).
    • Focus Order: Must be logical (usually follows DOM order).
    • Interaction: Native elements have standard key bindings (Enter/Space). Custom controls require JS implementation.
    • tabindex: Use 0 to make non-native elements focusable, -1 for programmatic focus only. Avoid positive values.
  • Introduction to ARIA:
    • Purpose: Enhances accessibility for dynamic content/custom widgets when native HTML is insufficient.
    • 5 Rules: Use native HTML first, don't change native semantics, ensure keyboard access, don't hide focusables, provide accessible names.
    • Categories: Roles (role="button"), Properties (aria-label, aria-labelledby, aria-describedby), States (aria-expanded, aria-hidden).
    • Use ARIA carefully and test thoroughly.
